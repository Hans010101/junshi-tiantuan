原项目适合作为产品概念原型，不适合直接公开上线。两份规划文档对产品价值、用户流程和商业化路径描述完整，但代码只实现了本地演示，文档中的 AI、云端、账号、报告、支付和运营能力并未真正落地。

旧工程已经可恢复地归档到 `/tmp/junshi-tiantuan-legacy-20260718`；两份原始 Word 文档保留在 `docs/reference/`。

## 已发现的问题

| 范围 | 发现 | 影响 |
| --- | --- | --- |
| AI | 使用关键词匹配与固定模板，没有模型调用 | 输出重复，不能处理真实复杂问题 |
| 安全 | 医疗急症、违法规避等输入会进入普通建议 | 高风险问题可能给出不恰当回答 |
| Web | 用户问题经 `innerHTML` 注入报告 | 存在跨站脚本风险 |
| 桌面端 | Web 固定为约 414px 手机外壳 | 不是真正的桌面界面 |
| 双端 | 小程序与 Web 的流程和能力不一致 | 维护成本高、体验分裂 |
| 数据 | 只有零散本地存储，没有真实历史/账号/同步 | 与规划文档不一致 |
| 云端 | 无后端、数据库、鉴权、监控或部署配置 | 无法稳定上线 |
| 文档 | Word 在无中文字体环境渲染时出现缺字 | 跨平台交付存在字体兼容问题 |

## 保留的产品资产

- 核心定位：结构化人物方法论，而非普通聊天机器人。
- 主流程：提出问题 → 匹配/选择视角 → 分析分歧 → 形成报告。
- 差异化方向：方法论库、视角匹配、深度追问、决策报告。
- 视觉气质：东方智囊意象与现代产品表达结合。

## 本次重建原则

1. 先用单一响应式 H5 覆盖 Web、手机和微信内置浏览器，避免维护两套前端。
2. 先不上账号和数据库，默认本机存储，降低成本与隐私风险。
3. AI 失败时返回基础分析，不让产品流程中断。
4. 在业务增长前就建立输入边界、专业风险提示和最小化日志。
5. 代码公开、部署可重复，生产环境由 Git 提交自动更新。
